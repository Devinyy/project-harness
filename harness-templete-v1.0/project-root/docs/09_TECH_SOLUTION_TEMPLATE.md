## 1. 背景与目标

### 1.1 需求背景

说明为什么要做这个需求，当前问题是什么，业务希望达成什么结果。

示例关注点：

- 当前流程有哪些痛点
- 现状是否存在效率问题、准确率问题、体验问题
- 是否属于已有能力扩展，还是全新模块

### 1.2 目标

明确本次前端实现要达成的结果：

- 用户能完成什么关键动作
- 页面或模块要具备什么能力
- 前端侧必须支持哪些业务规则

### 1.3 非目标

明确本次不做什么，避免范围失控。

示例：

- 本期不改移动端
- 本期不重构旧页面全部样式
- 本期不统一历史接口结构
- 本期不处理导出能力

## 11. 风险点与兼容性评估

### 11.1 业务风险

- 需求规则不明确
- 角色差异大
- 历史数据异常
- 流程涉及多状态跳转

### 11.2 技术风险

- 依赖老组件，改动有连锁影响
- 路由与缓存耦合
- 接口响应不稳定
- 页面状态复杂，容易出现脏数据
- 历史代码缺乏测试

### 11.3 兼容性

- 浏览器兼容
- 响应式 / 多端兼容
- 老数据兼容
- 历史入口兼容
- 权限模型兼容

### 11.4 回滚策略

- 代码层如何回滚
- 开关层如何兜底
- 新旧逻辑如何并行或切换
