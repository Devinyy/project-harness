## 1. 架构目标

当前项目的架构设计目标通常包括：

1. 将页面、业务逻辑、通用能力、基础设施分层
2. 让新增需求优先复用已有模块，而不是堆在页面里
3. 让接口接入、状态管理、权限处理有固定位置
4. 让新加入者能根据目录就推断“应该把什么放在哪里”
5. 控制公共区域的复杂度，降低重构和回归风险

请根据项目实际情况补充：

- 当前架构最关注的问题是什么
- 当前架构的历史包袱是什么
- 当前团队已经接受的折中是什么

---

## 2. 总体分层

推荐按以下层级理解应用：

1. **App / Shell 层**
   - 应用壳、Provider、全局初始化、全局守卫、布局外壳
2. **Route / Page 层**
   - 路由级页面、页面级数据请求、页面骨架、主流程编排
3. **Feature / Module 层**
   - 模块内业务组件、局部业务逻辑、局部状态与交互编排
4. **Shared / Domain 层**
   - 跨模块可复用的业务组件、hooks / composables、工具方法、领域模型
5. **UI / Base 层**
   - 基础组件、主题 token、样式原语、通用交互外壳
6. **Infrastructure 层**
   - 请求封装、日志、监控、权限接入、配置、构建脚本

> 一条经验：**越往下越稳定，越往上越贴近业务。**

## 3. 目录结构与职责

请用真实目录替换下面的示例：

```text
src/
├── app/
│   ├── providers/
│   ├── router/
│   ├── guards/
│   └── layouts/
├── pages/
│   ├── merchant/
│   ├── order/
│   └── settings/
├── features/
│   ├── merchant/
│   ├── order/
│   └── common/
├── shared/
│   ├── ui/
│   ├── business/
│   ├── hooks/
│   ├── utils/
│   ├── constants/
│   └── types/
├── services/
│   ├── merchant/
│   ├── order/
│   └── common/
├── store/
├── styles/
└── assets/
```

### 3.1 `app/`

职责：

- 应用初始化
- 全局 Provider / 插件注册
- 根布局
- 路由守卫
- 主题、国际化、监控、认证上下文注入

不应放入：

- 具体业务模块逻辑
- 局部页面专用组件
- 大量模块级状态逻辑

### 3.2 `pages/`

职责：

- 路由级页面容器
- 组合 Feature 组件
- 组织首屏数据请求
- 决定页面布局与主流程

不应放入：

- 大量可复用组件实现
- 复杂字段转换工具
- 公共 API 封装

### 3.3 `features/`

职责：

- 模块内业务组件
- 模块内表单、筛选栏、详情块、操作栏
- 局部 hooks / composables
- 与模块强相关的状态衍生和交互编排

适合放这里的内容：

- `MerchantFilter`
- `OrderStatusBar`
- `CampaignPublishDialog`
- `useMerchantTableColumns`

### 3.4 `shared/`

职责：

- 跨模块复用能力
- 通用 UI 基础组件
- 通用业务组件
- hooks / composables、utils、types、constants

放入标准：

- 至少被多个模块复用
- 与单一业务模块解耦
- 职责稳定、语义清晰

### 3.5 `services/`

职责：

- 按领域组织接口
- 隔离请求细节
- 统一处理请求/响应模型、错误、适配和重试策略

禁止：

- 在页面中散落裸请求
- service 层返回未经说明的随意结构
- 在 service 层写页面 UI 逻辑

### 3.6 `store/`

职责：

- 管理跨页面共享状态、缓存型状态、全局上下文
- 沉淀会被多个模块依赖的可观察状态

不建议：

- 把所有表单状态都塞进全局 store
- 把只在单组件使用的一次性状态提升到全局

## 4. 路由组织方式

请说明项目使用哪种路由组织方式：

- 集中式配置
- 文件式路由
- 动态路由注册
- 微前端子应用挂载
- 混合模式

### 4.1 路由定义位置

- 路由主配置文件：`<path>`
- 路由元信息：`<path>`
- 菜单与路由映射：`<path>`
- 权限拦截：`<path>`

### 4.2 路由设计原则

- 页面路径与业务语义保持一致
- 不在 URL 中暴露无意义实现细节
- 列表与详情的跳转路径有稳定模式
- 对于筛选条件，明确哪些保存在 URL，哪些保存在 local state / store
- 返回列表时尽量保留用户上下文（筛选条件、分页、滚动位置）

### 4.3 路由状态来源

建议明确：

- 哪些参数来自 path params
- 哪些来自 query params
- 哪些来自页面内状态
- 哪些来自 store

## 5. 数据流

推荐写出项目主数据流的默认模式，例如：

```text
User Action
  -> Page / Feature
  -> service
  -> adapter / mapper
  -> store / local state
  -> component render
  -> feedback / analytics / refresh
```

### 5.1 默认链路

建议说明：

- 谁负责触发请求
- service 返回的结构是否需要适配
- store 的职责是缓存、共享还是派生
- 组件应该接收什么粒度的数据
- UI 层负责哪些格式化展示

### 5.2 常见状态归属

请明确以下状态应该放哪里：

- 路由参数：URL
- 列表筛选条件：`<URL / page local state / store>`
- 表格分页：`<URL / page local state>`
- 表单临时输入：local state / form library
- 跨页面共享上下文：store
- 用户信息 / 权限：全局 store / context
- Feature Flag：全局配置层

### 5.3 刷新策略

需明确：

- 哪些操作成功后需要刷新列表
- 哪些场景做局部更新即可
- 是否允许乐观更新
- 是否有缓存失效策略

## 6. 状态管理边界

### 6.1 local state 适用场景

- 单页面内短生命周期状态
- UI 展开折叠
- 临时筛选输入
- 弹层开关
- 一次性表单草稿

### 6.2 store 适用场景

- 跨页面共享
- 跨组件层级共享
- 缓存型数据
- 全局上下文
- 高频被多个页面依赖的领域状态

### 6.3 禁止事项

- 因为“以后可能会用到”就提前提升状态
- 在 store 中保存难以追踪的一次性 UI 状态
- 页面、store、service 同时重复保存同一份真值而不做约束

## 7. API 与请求链路

### 7.1 请求入口

明确：

- 使用哪一层封装 HTTP client
- 超时、重试、鉴权、错误上报在哪里处理
- 上传下载如何处理
- 多租户、trace id、locale 等上下文如何注入

### 7.2 service 组织方式

推荐按业务域组织，例如：

```text
services/
├── merchant/
│   ├── query.ts
│   ├── mutation.ts
│   └── adapter.ts
├── order/
└── common/
```

或：

```text
services/
├── merchant.service.ts
├── order.service.ts
└── auth.service.ts
```

只要保证：

- 命名清晰
- 同一领域的请求放在一起
- 响应结构和适配逻辑有固定归属

### 7.3 适配层

建议明确：

- 接口原始字段是否直接进入 UI
- 若需要转换，由谁做
- 时间、金额、状态枚举是否统一在 adapter 处理
- null / undefined / 空数组的兜底逻辑放哪里

## 8. 权限、埋点、国际化

### 8.1 权限

说明：

- 页面访问权限在哪里判断
- 按钮权限在哪里判断
- 字段级权限有没有统一能力
- 无权限时展示什么

### 8.2 埋点

说明：

- 埋点入口统一在哪里
- 页面曝光与点击事件的组织方式
- 埋点参数是否有命名规范
- 是否要求与页面 / 组件解耦

### 8.3 国际化

如果项目需要 i18n，请说明：

- 文案资源文件路径
- 命名规则
- 何时允许直接写中文
- 异步加载、命名空间与回退策略

- 本地 mock、测试环境与线上差异如何兜底
- 配置变化是否需要重启或重新打包

## 11. 扩展规则

当新增需求时，优先按以下顺序扩展：

1. 复用现有页面模式
2. 复用已有 Feature 组件
3. 复用 shared 层的业务组件
4. 复用基础 UI 组件
5. 必要时新增局部组件
6. 只有在确实具备稳定复用价值时，才新增 shared / global 资产

### 11.1 新增页面

新增页面时应先回答：

- 路由放在哪里
- 使用哪种页面骨架
- 是否已有可复用表格 / 表单 / 筛选模式
- 请求和状态如何组织
- 是否影响权限与菜单
- 是否需要技术方案

### 11.2 新增通用能力

新增 shared 组件或全局能力前必须确认：

- 复用场景是否真实存在
- 命名是否不带单模块语义
- 是否会反向影响现有调用方
- 是否需要兼容旧逻辑

## 12. 反模式清单

以下做法应尽量避免：

- 在页面组件中直接书写大段请求、转换、缓存逻辑
- 为了局部需求直接修改全局基础组件默认行为
- 把单模块逻辑提前放进 shared
- 多个模块复制同一套表单 / 列表 / 状态标签代码
- 在多个层级保存同一份状态真值
- 用“临时方案”长期替代正式分层

---

## 13. 架构评审清单

新增功能或重构时，至少检查以下问题：

- 改动是否符合当前目录职责
- 页面、Feature、shared、service、store 的边界是否清晰
- 是否复用了现有模式
- 是否引入了新的公共抽象
- 是否影响权限、路由、埋点、国际化
- 是否动到了危险区域
- 是否需要同步更新 docs

---

## 14. 注意事项

基于本项目架构修改代码时，应注意：

- 先根据目录判断代码归属，再创建文件
- 遇到“放哪里都行”的情况，优先贴近业务、延迟抽象
- 不要因为追求“优雅”而重构大面积历史代码
- 不要将局部页面需求上升为全局基础设施修改
- 不要跨层直接耦合，例如页面直接改全局网络拦截器
