## 1. 总体原则

所有代码默认遵守以下原则：

1. **可读优先于炫技**
2. **一致性优先于局部个人偏好**
3. **显式优先于隐式**
4. **类型明确优先于“能跑就行”**
5. **复用优先于复制**
6. **小步扩展优先于大范围重构**
7. **贴近当前项目约束，不强行套入外部最佳实践**

## 2. 命名规范

### 2.1 通用命名原则

- 名称要体现职责，而不是实现细节
- 不使用模糊命名，如 `data`、`info`、`list2`、`temp`
- 布尔值命名要能读成句子，如 `isVisible`、`hasPermission`
- 事件处理函数使用动作前缀，如 `handleSubmit`、`onConfirm`
- 不使用无业务语义的缩写，除非是团队普遍接受的标准缩写

### 2.2 文件命名

请根据项目实际风格固定一套规则，例如：

- 组件文件：`PascalCase.tsx / .vue`
- hooks / composables：`useXxx.ts`
- 工具函数：`camelCase.ts` 或 `kebab-case.ts`
- 常量文件：`constants.ts` / `xxx.constants.ts`
- 类型文件：`types.ts` / `xxx.types.ts`
- service：`xxx.service.ts` / `xxx.api.ts`

> 不允许同一项目中混杂多套命名风格而没有说明。

### 2.3 变量 / 常量 / 类型

- 变量：`camelCase`
- 常量：`UPPER_SNAKE_CASE`
- 组件名 / 类型名 / 接口名：`PascalCase`
- 枚举值：`UPPER_SNAKE_CASE` 或与后端契约保持一致
- 布尔值：`is / has / can / should`
- 事件处理：`handleXxx`
- 计算属性 / 选择器：`get / select / computed`

### 2.4 反例

不推荐：

- `obj`, `res`, `arr`, `item2`, `newData`
- `flag` 但不说明是什么 flag
- `statusText1`
- `handleClick` 用于所有点击事件而不体现语义

## 3. TypeScript 规范

### 3.1 默认要求

- 尽量开启严格模式
- 不允许无理由使用 `any`
- 公共函数、service、hooks / composables、组件 props 必须有清晰类型
- 外部接口数据必须显式建模，不要一路透传 `unknown` 结构到 UI

### 3.2 何时允许 `any`

仅在以下情况可临时使用，并需附注释说明后续处理计划：

- 第三方库类型缺失，且短期无法补全
- 迁移历史代码的过渡阶段
- 一次性实验性代码且不会进入正式主干

不允许：

- 为省事绕过类型系统
- 在公共工具、service、store 中大面积使用 `any`

### 3.3 接口与类型建模

建议：

- API 原始类型与 UI 展示类型分离
- 枚举、联合类型、字面量类型优先于魔法字符串
- 复杂对象使用命名类型，不要大量内联匿名对象
- 对可空字段显式声明 `null | undefined` 语义

### 3.4 类型放置规则

- 组件私有类型：与组件同目录
- 模块共享类型：放到模块级 `types.ts`
- 全局通用类型：放到 `shared/types`
- API DTO：与 service 相邻，避免散落

## 4. 函数与模块设计

### 4.1 函数设计

- 函数应只做一类事情
- 控制参数个数，超过 4 个建议改为对象参数
- 避免过深的嵌套与过长函数
- 对副作用函数、纯函数做明确区分
- 复杂分支逻辑应拆到命名函数，而不是塞在 JSX / template 中

### 4.2 模块设计

- 一个文件聚焦一个核心职责
- 不要把请求、适配、组件、样式、常量全部塞进同一文件
- 模块间依赖方向应稳定，避免循环依赖
- 共享模块不依赖页面模块

### 4.3 导出规范

- 默认导出还是命名导出应统一
- barrel file (`index.ts`) 仅在不会制造隐式耦合时使用
- 不要为了“导出方便”把内部实现全部暴露

## 5. 组件代码规范

### 5.1 组件职责

- 页面组件负责流程编排
- 业务组件负责当前领域交互
- 基础组件负责通用 UI 能力
- 不要让组件同时承担 UI、业务规则、请求编排和埋点的全部职责

### 5.2 props / emits / events

- props 命名要体现语义
- 对回调参数给出明确类型
- 不要通过大量布尔开关堆出“万能组件”
- 多个互斥行为应考虑拆分组件，而不是继续增加 props

### 5.3 组件内逻辑组织

推荐顺序：

1. imports
2. 类型定义
3. 常量
4. component / setup
5. state
6. derived state
7. effects / lifecycle
8. handlers
9. render / template

### 5.4 JSX / Template 规则

- 避免把复杂逻辑直接写进模板
- 避免多层三元表达式
- 显式处理空态、加载态、错误态
- 超长区域拆成子组件或命名渲染函数

## 6. hooks / composables 规范

- hooks / composables 只封装可复用逻辑，不包装一次性流程
- 名称使用 `useXxx`
- 输出值结构保持稳定
- 不要在 hook 中偷偷修改全局状态而不暴露语义
- 需要副作用时，必须说明输入、输出、依赖和清理时机

建议结构：

```ts
type UseXxxOptions = {}
type UseXxxResult = {}

export function useXxx(options: UseXxxOptions): UseXxxResult {
  // ...
}
```

## 7. service / API 代码规范

- 页面与组件不要直接写裸请求
- service 名称与领域一致
- 请求参数类型、响应类型、适配逻辑应清晰分层
- 时间、金额、状态字段的转换不应散落在多个页面重复实现
- 对失败、空值、异常码要有统一约定

详细规则请与 `07_API_CONTRACTS.md` 保持一致。

## 8. 错误处理规范

### 8.1 原则

- 错误应被显式识别，而不是静默吞掉
- 用户可感知的错误要给出合适反馈
- 开发者需要排查的错误要保留上下文日志
- 业务错误、权限错误、网络错误、程序异常要尽量区分

### 8.2 禁止事项

- `catch` 后什么都不做
- 所有错误都统一提示“操作失败”
- 在 UI 层重复拼接后端错误逻辑
- 使用 `console.log` 代替正式错误上报

### 8.3 建议

- 提示文案可读、可行动
- 高风险操作失败时保留重试或回退路径
- 列表、详情、表单的错误态表现保持一致

## 9. 样式规范

### 9.1 总体原则

- 样式应与组件职责保持一致
- 优先使用项目既有样式体系
- 不为局部需求直接污染全局样式
- 主题、token、间距、字号、色值优先复用设计系统定义

### 9.2 约束

- 不直接硬编码大量魔法数值
- 不复制粘贴整个页面样式块
- 不在公共组件中写死业务特定尺寸与色彩
- 不绕过主题 token 直接使用随意十六进制色值（若项目有 token 体系）

### 9.3 响应式与兼容性

请补充：

- 断点策略
- 是否优先 PC / Mobile
- 滚动容器规则
- 字体与浏览器兼容要求

---

## 10. 导入与依赖规范

- 导入顺序固定，例如：第三方 -> 别名模块 -> 相对路径 -> 样式
- 禁止跨层级深层相对路径引用导致耦合失控
- 尽量使用统一路径别名
- 新增依赖前确认是否已有现有方案可复用
- 公共依赖升级要评估影响面

---

## 11. 注释与文档规范

- 注释用于解释“为什么”，不是重复“做了什么”
- 复杂业务规则、状态流转、兼容逻辑必须有解释
- 临时兼容代码必须标记清楚移除条件
- 公共函数、危险逻辑、复杂 hook 推荐保留简短说明

不推荐：

- 注释与代码明显不一致
- 用注释掩盖坏代码而不治理
- 大量保留过时 TODO 而无人维护

---

## 12. 测试与自检

根据项目现状，至少说明：

- 哪些模块要求单测
- 哪些流程要求手测
- 哪些页面要求截图 / 像素回归 / E2E
- 哪些 bug 修复必须附回归点

建议每次提交前完成：

- 类型检查
- lint
- 核心路径手测
- 异常态和边界态验证
- 权限 / 空态 / loading / 成功提示验证

---

## 13. Performance / 可访问性 / 国际化

如果项目要求，请明确：

- 哪些场景需要懒加载、虚拟列表、缓存优化
- 哪些组件必须支持键盘操作与语义标签
- 文案是否必须走 i18n
- 日期、金额、时区、数字格式如何统一

---

## 14. 禁止清单

以下行为原则上禁止：

- 页面中直接裸调接口
- 通过修改全局基础组件解决局部需求
- 在 render / template 内进行复杂数据改造
- 使用 `any` 绕过关键类型建模
- 把业务规则散落在多个按钮点击函数中
- 复制粘贴相似页面后不抽公共逻辑
- 把一次性需求强行抽象成全局万能组件
- 在未知影响面的情况下修改危险区域

## 16. 建议与工具配置对齐

建议把以下规则尽量工具化：

- ESLint：命名、import 顺序、禁止 unused、禁止 console 等
- Prettier：格式统一
- TypeScript：strict / noImplicitAny / path alias
- Stylelint：样式约束
- Husky / lint-staged：提交前校验
- Commitlint：提交规范

文档规则 != 工具规则。
最理想的状态是：**能自动检查的尽量自动检查，不能自动检查的再通过文档约束。**
