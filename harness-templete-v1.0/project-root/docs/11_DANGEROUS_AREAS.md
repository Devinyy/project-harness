> - 若项目中存在真实高风险目录，请写出具体路径和 owner
> - 对高风险区域的改动，建议附更完整的测试与回滚说明

## 1. 风险等级定义

### P0：关键基础设施

一旦改错，可能导致：

- 应用无法启动
- 大面积白屏
- 登录失效
- 全站请求异常
- 权限系统失效
- 全局样式错乱
- 大面积数据异常

### P1：高影响公共模块

一旦改错，可能导致：

- 多个页面共同异常
- 大量交互模式变化
- 某业务域普遍回归
- 核心路径性能下降

### P2：中风险共享模块

一旦改错，影响若干模块，但一般可局部回滚。

## 2. 常见危险区总览

请根据项目实际情况补充真实目录路径和负责人。

### 2.1 应用壳层 / 根入口（P0）

典型位置：

- `src/main.ts`
- `src/App.tsx`
- `src/app/*`
- `layouts/*`

风险：

- 应用无法挂载
- Provider 顺序错乱
- 国际化 / 主题 / 路由 / 权限上下文失效

### 2.2 路由系统（P0）

典型位置：

- `router/*`
- `routes/*`
- 路由守卫
- 菜单与路由映射

风险：

- 页面无法访问
- 菜单高亮异常
- 动态路由失效
- 参数丢失
- 返回路径与上下文恢复异常

### 2.3 认证与权限（P0）

典型位置：

- Token 管理
- 登录态恢复
- 权限守卫
- 按钮权限 / 字段权限工具

风险：

- 越权访问
- 全量拦截导致误拒绝
- 登录循环跳转
- 权限缓存脏读

### 2.4 HTTP client / 拦截器（P0）

典型位置：

- 请求封装
- request / response interceptor
- 错误统一处理
- trace、tenant、locale 注入

风险：

- 全站请求失败
- Token 失效处理错误
- 重试风暴
- 下载上传异常
- 错误提示泛滥

### 2.5 全局状态 / 用户上下文（P0 / P1）

典型位置：

- user store
- auth store
- app config store
- feature flags

风险：

- 用户上下文丢失
- 权限信息错乱
- 环境配置失真
- 页面共享状态异常

### 2.6 基础 UI 组件（P1）

典型位置：

- `shared/ui/*`
- `components/base/*`

风险：

- 大量页面样式或交互回归
- 表单、表格、弹层默认行为被破坏
- 局部需求污染全局

### 2.7 全局样式 / token / 主题（P0 / P1）

典型位置：

- `styles/global.*`
- design tokens
- 主题变量
- reset 样式

风险：

- 全站样式错位
- 字体、字号、间距异常
- 暗色模式 / 多主题失效
- 组件视觉统一性被破坏

### 2.8 构建与运行配置（P0）

典型位置：

- `vite.config.*`
- `webpack.config.*`
- `next.config.*`
- `tsconfig.*`
- `package.json`
- `env*`

风险：

- 本地或 CI 构建失败
- 打包产物异常
- 动态导入失效
- alias 解析错误
- 环境变量注入错误

### 2.9 埋点 / 监控 / 日志（P1）

风险：

- 关键事件丢失
- 监控误报或漏报
- 线上排障难度大幅上升

### 2.10 国际化 / 文案资源（P1）

风险：

- 文案找不到 key
- 默认语言回退异常
- 局部页面报错或显示 key 名

## 3. 修改危险区前必须做的事

在改动上述区域前，至少完成：

1. 说清楚为什么必须改这里
2. 盘点影响范围
3. 查找调用方 / 依赖方
4. 确认是否存在更局部的替代方案
5. 写出回滚方式
6. 补齐最低限度自测计划

对 P0 改动，建议额外要求：

- 至少一位熟悉该模块的 reviewer
- 必要时先出技术方案
- 明确灰度或回滚策略

---

## 4. 典型红线

以下操作属于高风险红线，除非充分评估，否则不要做：

- 为满足单页面需求直接改基础 Button / Table / Modal 默认行为
- 在全局请求拦截器中追加局部业务逻辑
- 随意修改 Token 读取、存储、刷新逻辑
- 在未盘点调用方的情况下修改公共组件 props 语义
- 随意删改全局样式 reset
- 在生产构建配置中直接调整 split / alias / define 等关键项
- 在权限工具中加入项目外特例而不做隔离
- 修改路由路径或 query 协议却不排查历史外链与收藏夹兼容性

## 6. 不同危险区的专项检查

### 6.1 改路由前

检查：

- 菜单是否受影响
- 面包屑是否受影响
- query 参数是否兼容
- 权限守卫是否匹配
- 返回列表的上下文是否丢失
- 是否有历史链接依赖

### 6.2 改权限前

检查：

- 页面权限和按钮权限是否同时变化
- 用户缓存是否需要刷新
- 超级管理员 / 特殊角色是否受影响
- 无权限时 UI 表现是否合理

### 6.3 改 HTTP client 前

检查：

- 鉴权逻辑
- 错误码处理
- 上传 / 下载
- 超时与重试
- 多租户 / locale / trace 注入
- 失败埋点 / 监控

### 6.4 改基础组件前

检查：

- 当前组件被多少页面使用
- 默认 props 是否变化
- DOM 结构是否影响自动化测试或样式覆盖
- 是否会引发视觉差异
- 是否应该新增变体而不是直接改默认值

### 6.5 改构建配置前

检查：

- 本地 dev 是否正常
- CI 是否正常
- 生产构建是否正常
- 产物路径、hash、动态资源加载是否正常
- Source Map、监控上传是否正常

## 7. 回滚建议

对危险区改动，推荐至少满足一种回滚能力：

- Git revert 可快速回退
- Feature Flag 可关闭
- 新老逻辑可切换
- 公共组件支持兼容旧参数
- 构建配置变更可快速恢复

若无法快速回滚，应在方案和评审中明确说明。

---

## 8. 回归测试建议

对危险区改动，不应只测“我改的页面”。

建议覆盖：

- 登录 / 登出
- 首页进入
- 关键业务主路径
- 典型列表页 / 详情页 / 表单页
- 不同角色
- 异常态
- 构建 / 部署 / 线上监控指标

## 10. 关键词警戒清单

当改动涉及以下关键词时，应默认提高警惕并先查阅本文件：

- route / router / menu / guard
- auth / token / permission
- interceptor / http client
- shared ui / base component
- global style / theme / token
- build / config / alias / env
- provider / app shell / layout

避免以下做法：

- 为满足局部页面需求直接重构全局能力
- 未盘点影响面就修改公共默认行为
- 未写回滚说明就动 P0 区域
- 未验证不同角色和主路径就提交危险区改动
