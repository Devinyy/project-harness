## 1. 文档目标

本文件不讲抽象原则，而讲“拿到任务后具体怎么做”。

它适合回答以下问题：

- 新增列表页先做什么
- 新增详情页怎么拆
- 接一个新接口要改哪些地方
- 什么时候需要 store
- 加一个权限点通常会影响哪里
- 修一个 bug 不应只改表面现象
- 什么时候必须先写技术方案

---

## 2. 通用工作流

无论什么任务，默认先做以下 6 步：

1. 明确目标：用户要完成什么动作
2. 查文档：阅读相关 docs
3. 查现有实现：寻找同类页面 / 组件 / service / store
4. 判断范围：确认是否要出技术方案
5. 决定改动点：列出文件和风险
6. 开发 + 自测 + 更新文档

### 2.1 开始前最少要回答的问题

- 改动入口在哪里
- 是否已有同类实现
- 是否涉及新路由、新接口、新权限、新埋点
- 是否动到公共组件或全局逻辑
- 成功后的自测点有哪些

### 2.2 标准输出格式（建议 AI 使用）

在正式改代码前，先列出：

- 任务目标：
- 主要改动文件：
- 复用点：
- 风险点：
- 自测点：
- 是否需要技术方案：是 / 否

## 3. Playbook A：新增列表页

### 3.1 适用场景

- 新增 B 端后台列表页
- 带筛选、分页、表格、批量操作
- 可能附带详情抽屉 / 弹窗

### 3.2 开始前要看

1. `02_ARCHITECTURE.md`
2. `05_COMPONENT_PATTERNS.md`
3. `06_UI_COMPONENT_GUIDE.md`
4. `07_API_CONTRACTS.md`
5. `11_DANGEROUS_AREAS.md`

### 3.3 标准步骤

1. 找到最相似的现有列表页
2. 确定路由和菜单是否需要新增
3. 确定筛选条件、分页、排序放哪里
4. 确定表格列定义如何组织
5. 明确数据请求由 page 还是 query hook 发起
6. 明确详情查看 / 行操作 / 批量操作模式
7. 明确成功后刷新策略
8. 补空态、loading、错误态、无权限态

### 3.4 最少产物

- 页面组件
- 筛选组件
- 表格列定义
- service / query hook
- 必要的状态标签或行操作组件
- 路由 / 菜单（如需要）

### 3.5 容易漏掉的点

- 返回列表时是否恢复筛选和分页
- 删除当前页最后一条后的页码处理
- 列表权限与按钮权限是否一致
- 表格空态文案是否合理
- 批量操作是否有上限与禁用条件

## 4. Playbook B：新增详情页

### 4.1 适用场景

- 独立详情页
- 列表联动详情
- 只读 / 编辑切换详情

### 4.2 标准步骤

1. 明确详情入口来自哪里
2. 明确详情数据与列表数据的关系
3. 确定首屏需要展示哪些 summary 字段
4. 确定 section 如何拆分
5. 明确操作按钮与状态限制
6. 确定返回路径与上下文恢复
7. 处理空态 / 无权限 / 资源不存在

### 4.3 容易漏掉的点

- 列表数据不能替代详情真值
- 详情页按钮权限可能与列表行操作不同
- 某些字段为空值展示要有兜底
- 状态切换后详情和列表是否都要刷新

---

## 5. Playbook C：新增创建 / 编辑表单

### 5.1 适用场景

- 新建页面
- 编辑页面
- 弹窗表单 / 抽屉表单

### 5.2 标准步骤

1. 确定是页面表单还是弹层表单
2. 找最相似表单模式复用
3. 明确字段分组与顺序
4. 明确默认值来源
5. 明确同步校验与异步校验
6. 明确草稿态、回填、重置、提交逻辑
7. 明确提交成功后的跳转或刷新策略
8. 明确失败后的错误反馈位置

### 5.3 容易漏掉的点

- 编辑态字段是否全部可改
- 某些字段是否只在新建态可见
- 后端字段错误如何映射到表单项
- 提交中是否防重复点击
- 关闭弹层或离开页面是否需要二次确认

---

## 6. Playbook D：接入新接口 / 替换旧接口

### 6.1 适用场景

- 新增查询接口
- 新增提交接口
- BFF 替换旧接口
- 接口字段变更

### 6.2 标准步骤

1. 查现有 service 目录与同领域命名
2. 明确请求参数、响应结构、错误码
3. 判断是否需要 adapter
4. 决定请求状态谁管理
5. 明确缓存 / 刷新 / 重试 / 取消策略
6. 确定空值、时间、金额、状态映射
7. 增加错误态和边界态验证
8. 更新相关文档或示例

### 6.3 容易漏掉的点

- 分页协议页码起始不一致
- 枚举值和展示文案未统一
- 401 / 403 / 404 的展示混淆
- 删除或更新后缓存未失效
- API 改了但表格列、筛选项、导出参数没一起改

---

## 7. Playbook E：新增 store / 共享状态

### 7.1 使用前先问

- 这份状态是否跨页面 / 跨模块共享
- 是否需要缓存
- 是否需要被多个组件订阅
- 是否真的不能用 local state / query cache 解决

### 7.2 标准步骤

1. 先确认新增 store 的理由
2. 明确真值、派生值、写入入口
3. 确定读取方有哪些
4. 明确初始化、重置、失效时机
5. 明确与 URL、form state、query cache 的边界
6. 自测跨页面与回退场景

### 7.3 容易漏掉的点

- store 中残留旧页面状态
- 多个地方都能写入导致难排查
- 与 URL 参数双向同步时发生竞态
- 只在单一页面使用却仍提升到全局

---

## 8. Playbook F：新增权限 / 菜单 / 路由

### 8.1 标准步骤

1. 明确是页面权限、按钮权限还是数据权限
2. 确认路由和菜单元信息如何配置
3. 确认权限判断位置
4. 无权限时展示什么
5. 确认埋点和审计是否受影响
6. 验证不同角色场景

### 8.2 容易漏掉的点

- 菜单可见但页面打不开
- 页面能打开但按钮越权
- 权限缓存未刷新
- 字段可见性和操作权限未联动
- 超级管理员的特殊逻辑污染普通角色流程

## 9. Playbook G：修复 Bug

### 9.1 原则

修 bug 不是“哪里报错改哪里”，而是先找根因。

### 9.2 标准步骤

1. 复现问题
2. 明确触发条件、角色、环境、页面路径
3. 判断问题属于 UI、状态、API、权限还是配置
4. 查日志、接口、状态、历史提交
5. 先修根因，再看是否需要补保护逻辑
6. 回归相关路径，而不是只验证一个按钮

### 9.3 Bug 修复说明建议包含

- 根因
- 影响范围
- 修复点
- 回归点
- 是否需要补文档 / 排障手册

---

## 10. Playbook H：重构组件 / 公共模块

### 10.1 适用场景

- 公共组件优化
- 目录治理
- 数据流调整
- 接口层重构

### 10.2 重构前必须确认

- 是否有真实收益
- 是否能小步提交
- 是否会影响多个调用方
- 是否触及危险区域
- 是否需要灰度 / 回滚预案

### 10.3 标准步骤

1. 明确重构目标与非目标
2. 盘点调用方
3. 决定兼容策略
4. 小步迁移
5. 增量回归
6. 更新 docs 和 examples

---

## 11. Playbook I：开发前输出技术方案

当任务满足以下任一条件时，应先出方案：

- 新增页面或完整模块
- 同时影响 route / page / feature / service / store
- 涉及复杂状态流转
- 业务规则多且边界复杂
- 重构公共区域

执行方式：

1. 按 `10_TECH_SOLUTION_PLAYBOOK.md` 推导
2. 用 `09_TECH_SOLUTION_TEMPLATE.md` 输出
3. 评审通过后再正式开发

---

## 12. Playbook J：文档更新

满足以下情况时，开发结束后应同步更新 docs：

- 新增了可复用模式
- 修改了公共 API 契约
- 引入了新危险区
- 排查出了新的高频故障
- 技术方案模板出现了新的固定章节
- 业务领域词汇发生变化

建议同步更新：

- 对应主题文档
- `00_INDEX.md`（如目录变化）
- `examples/`（如新增一个好示例）

## 14. 最后提醒

Playbook 的目的不是限制创造力，而是减少下面这些低级损耗：

- 重复踩坑
- 忘记补空态 / loading / 权限
- 跳过现有模式直接新造轮子
- 动到公共区域却没有自检
- 复杂需求不出方案直接开写

当团队形成稳定套路后，协作效率才能真正提升，返工才能真正减少。
