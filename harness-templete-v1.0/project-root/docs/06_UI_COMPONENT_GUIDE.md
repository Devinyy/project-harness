## 1. 文档目标

本文档用于回答以下问题：

1. 当前项目的 UI 组件分几层，分别负责什么
2. 新需求应优先复用哪些组件，而不是重复造轮子
3. 常见页面、表单、表格、弹窗、反馈态应该怎么选型
4. 哪些组件可以扩展，哪些组件禁止直接改
5. 新增页面时，应该遵守哪些默认 UI 约定

> 本文档不是组件 API 手册，不要求逐个枚举所有 props。
> 本文档的重点是：**选型规则、组合模式、约束边界、默认做法**。

---

## 2. 使用原则（AI / 开发者通用）

### 2.1 决策优先级

在新增 UI 时，按以下优先级决策：

1. **优先复用现有业务组件**
2. 现有业务组件不满足时，**组合基础组件**
3. 基础组件无法满足时，**在 shared / ui 层扩展**
4. 只有在确实不存在合适抽象时，**新增组件**

### 2.2 禁止事项

禁止以下行为：

- 为了赶进度，绕开现有组件体系，直接在页面里堆原子样式
- 因为一个局部需求，直接修改全局基础组件的默认行为
- 在多个页面复制同一套表单、表格、状态徽标、筛选栏实现
- 在业务页面直接依赖第三方 UI 库原始组件，跳过项目封装层
- 组件命名和职责不清，既承载 UI，又承载大量业务编排逻辑

### 2.3 默认原则

- **一致性优先于局部“更好看”**
- **复用优先于一次性快捷实现**
- **组合优先于继承**
- **配置优先于复制**
- **可维护优先于短期最少代码**

## 3. UI 组件分层

> 请根据项目实际目录调整路径名称。

推荐分层如下：

### 3.1 基础展示层（`shared/ui` 或 `components/base`）

职责：

- 对第三方组件库做轻量封装
- 提供统一的 Button / Input / Select / Modal / Table / Tabs / Tag / Empty / Spinner 等基础能力
- 管理项目统一的尺寸、间距、状态色、危险操作风格、空态与加载态表现

特征：

- 无业务语义
- 尽量少依赖具体页面
- 可在全项目复用

示例：

- `AppButton`
- `AppInput`
- `AppSelect`
- `AppModal`
- `AppTable`
- `StatusTag`
- `PageContainer`
- `SectionCard`

### 3.2 业务通用层（`components/business` 或 `shared/business`）

职责：

- 沉淀在多个业务模块中重复出现的组合模式
- 将“业务语义 + UI 模式”封装为统一组件

示例：

- `SearchFilterBar`
- `UserPicker`
- `PermissionGuard`
- `OrderStatusTag`
- `AmountDisplay`
- `DateRangeFilter`
- `EntityTableToolbar`

### 3.3 模块内组件层（`features/*/components`）

职责：

- 服务某一个业务模块或页面
- 可以持有当前模块上下文，但不应污染全局

示例：

- `OrderListFilter`
- `MerchantDetailHeader`
- `CampaignPublishDialog`

### 3.4 页面层（`pages/*` 或 `routes/*`）

职责：

- 路由级容器
- 组织页面结构、发起数据请求、组合业务组件
- 不承载可复用的通用 UI 实现

页面层不要：

- 大量定义重复样式原子
- 重写通用弹窗、表格、筛选栏逻辑
- 直接堆砌过多匿名 render 逻辑

## 4. 组件选型规则

## 4.1 按钮（Button）

### 使用原则

- 主要操作：页面中最重要且建议用户执行的动作
- 次要操作：辅助动作
- 危险操作：删除、停用、关闭不可恢复流程等

### 默认规则

- 每个区域通常只允许 **1 个主按钮**
- 危险操作必须使用危险态，不允许与主要操作视觉同权
- 加载中的按钮要禁止重复点击
- 触发异步提交的按钮必须有 loading 状态

### 不推荐

- 同一区域并列多个主按钮
- 将“取消”做成高强调按钮
- 提交中仍允许连续点击

---

## 4.2 弹层：Modal vs Drawer

### 用 Modal 的场景

- 表单字段较少
- 用户完成一个短流程即可关闭
- 需要强打断
- 需要用户明确确认或取消

### 用 Drawer 的场景

- 表单较长
- 需要查看上下文，同时完成编辑
- 有较多分组内容
- 需要保留原页面信息

### 不推荐

- 长表单塞进小型 Modal
- 简单确认操作使用 Drawer
- 在一个页面同时打开多层嵌套弹窗

---

## 4.3 数据展示：Table vs List vs Card

### 用 Table 的场景

- 数据结构稳定
- 字段列清晰
- 用户需要排序、筛选、批量操作、分页
- 后台管理系统主列表页

### 用 List 的场景

- 内容结构不完全规则
- 强调阅读顺序
- 每项信息不适合严格列对齐

### 用 Card 的场景

- 单项信息块较强
- 视觉上强调分组或状态
- 移动端 / 看板型场景更合适

### 默认规则

- B 端后台列表页优先使用 Table
- 如果用户任务是“筛选 + 查看 + 操作”，通常先考虑 Table
- 如果用户任务是“浏览 + 理解内容”，可优先 List / Card

---

## 4.4 表单组件选择

### 输入型

- 短文本：Input
- 长文本：Textarea
- 数值：NumberInput / Input + formatter/parser
- 金额：统一金额输入与展示组件，不要手写格式化
- 日期：DatePicker / RangePicker
- 枚举少量：Radio / Segmented
- 枚举较多：Select
- 可搜索实体：SearchSelect / RemoteSelect / Picker
- 开关型：Switch（即时生效）或 Checkbox（提交后生效）

### 默认规则

- 能枚举的不要让用户手输
- 有稳定实体源的不要使用自由文本
- 涉及远程搜索时必须处理 loading、empty、error
- 金额、百分比、手机号、身份证、时间等特殊字段必须统一格式策略

---

## 4.5 反馈组件

### 必备反馈态

所有异步区域至少考虑以下反馈态：

- 初始 loading
- 空态
- 接口错误态
- 提交成功反馈
- 权限不足提示（如有权限控制）

### 推荐表现

- 页面级首屏加载：Skeleton 或 Page Spinner
- 局部加载：Section Spinner / Button Loading
- 空数据：统一 Empty 组件
- 非阻塞反馈：Toast / Message
- 强提示：Alert / Inline Error / Result 页面

---

## 5. 页面级 UI 规则

## 5.1 页面骨架

推荐页面结构：

1. PageHeader：标题、描述、主操作
2. FilterArea：筛选条件（如果是列表页）
3. ContentArea：表格 / 卡片 / 表单主体
4. SecondaryArea：说明、统计、附加信息
5. FooterAction：提交 / 返回 / 底部操作（如需要）

### 默认要求

- 页面标题必须清晰表达当前实体或任务
- 筛选区与结果区视觉上应清晰分层
- 操作区位置要稳定，避免每页都不一样
- 不要把说明文案塞进 placeholder 里代替正式提示

## 5.2 间距与密度

请按项目设计系统替换具体数值。默认建议：

- 页面区块间距：24
- 区块内元素垂直间距：16
- 表单项最小垂直间距：12
- 表单分组间距：24
- 列表筛选区与表格区之间间距：16 ~ 24

> 如项目已有 design token，以 token 为准，不要写死。

## 5.3 操作布局

- 主操作靠近标题或表单底部固定区域
- 次要操作与主操作分层展示
- 危险操作应远离主流程操作，并增加确认
- 表格行内操作不宜过多，超出 3 个优先折叠到更多菜单

## 6. 表单标准模式

## 6.1 表单默认要求

- 每个字段必须有明确 label
- 必填项规则明确，避免只依赖 placeholder
- 提交前要有前端基础校验
- 提交按钮需要 disabled / loading 保护
- 异步初始化时要考虑默认值回填
- 编辑态与新建态要明确区分

## 6.2 字段组织

优先按业务语义分组，而不是按组件类型堆叠：

- 基础信息
- 配置项
- 权限与范围
- 高级设置
- 风险操作区

## 6.3 错误处理

- 字段级错误：展示在字段附近
- 表单级错误：展示在表单顶部或提交区附近
- 服务端校验错误：尽量映射到具体字段
- 未知错误：统一错误提示，不暴露技术细节

## 6.4 只读态

只读态不要简单禁用整页表单控件，应优先考虑：

- 使用描述性展示组件
- 保留格式与单位
- 对敏感字段做脱敏处理

---

## 7. 表格与列表标准模式

## 7.1 列表页推荐结构

1. 页面标题
2. 筛选栏
3. 批量操作区（如有）
4. 结果表格 / 列表
5. 分页
6. 空态 / 错误态 / 无权限态

## 7.2 表格列设计

- 列标题使用业务术语，不使用内部字段名
- 金额、时间、状态、人数等常见列格式统一
- 操作列保持稳定，不要每行数量差异过大
- 对长文本使用省略与 Tooltip，但不要隐藏关键信息
- 状态优先使用状态标签，不要只靠纯文本

## 7.3 筛选栏要求

- 常用筛选默认展示
- 低频筛选可折叠到“更多筛选”
- Reset / Search 行为明确
- 筛选项命名与后端字段可不同，但必须语义清楚
- 远程下拉必须防抖与异常兜底

---

## 8. 组件封装约定

## 8.1 什么时候新建业务组件

满足以下任一条件时，优先提炼业务组件：

- 相同 UI + 相同业务语义在 2 个以上页面重复出现
- 组件内部需要统一处理权限、状态映射、文案格式
- 某个实体选择器、状态标签、筛选栏被多个模块复用

## 8.2 什么时候不要抽组件

以下情况不要过早抽象：

- 仅出现一次，且短期没有复用预期
- 只是两个字段顺序相似，但业务语义不同
- 抽象后 props 爆炸，反而难懂

## 8.3 封装要求

- 对外暴露稳定、少量、语义清晰的 props
- 避免把所有内部控制项都暴露出去
- 复杂业务组件要有最小可用文档或示例
- 组件内部如依赖接口，优先解耦为外部传参或 hooks

---

## 9. 样式约定

> 结合项目现有样式方案填写：CSS Modules / Tailwind / Less / SCSS / CSS-in-JS。

默认要求：

- 样式实现服从项目统一方案，不混用多套风格
- 避免在页面中堆积大量 inline style
- 不直接写魔法数字，优先使用 design tokens / 变量 / 语义类名
- 状态色、圆角、字号、阴影以设计系统为准
- 响应式规则保持一致，不自行发明断点

---

## 10. 无障碍、国际化与埋点

## 10.1 无障碍

- 可点击元素必须有清晰文本或 aria-label
- 表单控件需与 label 关联
- 禁止只靠颜色表达状态
- 键盘可达路径要合理

## 10.2 国际化

- 不在组件内部硬编码业务文案
- 时间、金额、数字、状态文案使用统一格式化方案
- 预留多语言场景时，组件命名与字段语义不要绑定中文文案

## 10.3 埋点

- 关键按钮、筛选、提交、导出、切换视图等操作需考虑埋点
- 不在视图层随意拼接埋点字段，统一从埋点层或常量管理

---

## 11. 反模式清单

以下是明确反模式：

- 页面里直接 copy 另一个页面的整段 JSX / Vue 模板
- 为了适配一个页面，修改全局 Button / Table 默认样式
- 状态标签在不同页面颜色和文案不一致
- 表单校验逻辑散落在多个 click handler 中
- 列表页没有 loading、empty、error 任一兜底
- 把“看起来可复用”的所有块都提前抽象成通用组件
- 在 UI 组件里写大量接口请求与业务判断，导致难以复用

---

## 12. 新增组件决策流程

新增页面或组件时，按以下顺序决策：

1. 先查找是否已有对应业务组件或页面模板
2. 没有现成组件时，再查基础 UI 层
3. 确认该需求属于页面、业务组件还是基础组件
4. 明确说明要复用哪些现有组件
5. 如果需要新增组件，解释为何无法复用现有组件
6. 对表单、表格、弹层、反馈态显式说明选型理由
7. 提交前检查是否违反本文件中的反模式

---

## 13. 新增组件前检查清单

在新增任意组件前，先回答以下问题：

- 是否已有同类组件可复用？
- 新组件属于基础层、业务通用层，还是模块内组件？
- 该组件是否真的会被复用？
- 是否会破坏现有样式或交互一致性？
- 是否需要支持 loading / empty / error / disabled / readonly？
- 是否需要权限控制、埋点、国际化、格式化能力？
- 是否已有示例可参考？
- 是否存在更轻的组合方式，而非新建组件？

## 15. 最终要求

在本项目中，UI 决策的默认原则是：

- 不猜测
- 不重复造轮子
- 不跳过封装层
- 不破坏一致性
- 先复用，再扩展，最后才新增

如与设计稿、业务需求或历史实现冲突，优先顺序为：

1. 已确认的业务需求
2. 本文档与项目现有约定
3. 设计系统与现有组件库
4. 个别页面的历史特例

当无法判断时，不要自行发明新模式，应在技术方案中显式说明不确定点。
