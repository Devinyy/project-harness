# agent.md

> 本文件是 AI agent 在本项目中的行为指令。
> 放置位置：项目根目录（与 `docs/` 同级）。
> 适用工具：Claude Code (CLAUDE.md) / Cursor (.cursorrules) / Windsurf / 其他 AI 编码助手。
> 本文件只给 AI 读，不替代 `docs/` 中的项目知识文档。

---

## 1. 核心原则

1. **先读，再写** — 先读相关文档，再决定改哪些目录和文件
2. **优先复用** — 先查现有页面、组件、hooks / composables、store、service
3. **不确定就标记** — 对术语、接口、权限、状态流转不确定时，标记为 `<!-- TODO: 待确认 -->`，不要猜测
4. **不要越层** — 不允许在页面层堆太多通用逻辑，不允许公共组件夹带业务特例
5. **不要越权设计** — 没有明确需求时，不自行引入新框架、新库、大规模重构
6. **变更后同步文档** — 当架构、约束、通用模式发生变化时，同步更新 docs/

### 2.1 每次会话必读

每次新会话开始时，先读 `docs/00_INDEX.md`，建立当前项目的文档全景。

### 2.2 按任务类型读取

根据任务类型，按以下映射阅读相关文档：

| 任务类型 | 必读文档 |
|---------|---------|
| 首次接手项目 | 00, 01, 02, 03, 04 |
| 新增列表页 | 02, 05, 06, 07, 08 |
| 新增详情页 | 02, 05, 06, 07, 08 |

> 也可查阅 `docs/manifest.json` 中的 `task_to_docs` 字段获取完整映射。

### 2.3 特定场景触发

- 遇到业务术语不确定 → 先查 `docs/03_BUSINESS_DOMAIN.md`

- 涉及路由 / 权限 / 拦截器 / 基础组件 / 全局样式 / 构建配置 → 先查 `docs/11_DANGEROUS_AREAS.md`

## 3. 任务路由

根据输入类型，执行对应的工作流：

```text
收到 PRD / 需求描述
  → 判断是否需要先出技术方案（参考 docs/10）
  → 是 → 执行 skill: 生成技术方案
  → 否 → 执行 skill: 编写代码

收到已定稿的技术方案
  → 执行 skill: 编写代码

代码开发完成
  → 执行 skill: 前端自测
```

## 4. 各领域行为约束

### 4.1 架构层

- 先根据目录判断代码归属，再创建文件
- 遇到"放哪里都行"的情况，优先贴近业务、延迟抽象
- 不要因为追求"优雅"而重构大面积历史代码
- 不要将局部页面需求上升为全局基础设施修改
- 不要跨层直接耦合，例如页面直接改全局网络拦截器

### 4.2 业务领域

- 修改业务逻辑前，必须先查 `docs/03_BUSINESS_DOMAIN.md`
- 不要看到英文枚举就自行推断业务含义
- 不要仅根据某个页面表现推断全局业务规则
- 不要将历史字段名当作推荐命名继续扩散
- 未确认就不要改动状态流转与权限判断

### 4.3 编码规范

- 遵循项目现有命名和目录风格，不引入项目中不存在的命名方式
- 输出的类型、组件、service、store 边界必须清晰
- 不要因为模型偏好而引入额外抽象层
- 不要自己发明项目中不存在的工具、hooks 或组件命名
- 不要把"猜测"写成确定实现

### 4.4 组件开发

- 优先寻找现有页面骨架和同类模块
- 优先延续已有命名与目录风格
- 不要默认把每个区块都抽成共享组件
- 不要把局部特例硬塞进公共组件
- 不要在 JSX / template 中堆大量业务逻辑
- 当组件边界不清时，先列出候选拆分方案，再选最贴近当前需求的一种

### 4.5 UI 选型

新增页面或组件时，按以下顺序决策：

1. 先查找是否已有对应业务组件或页面模板
2. 没有现成组件时，再查基础 UI 层
3. 确认该需求属于页面、业务组件还是基础组件
4. 明确说明要复用哪些现有组件
5. 如果需要新增组件，解释为何无法复用现有组件
6. 对表单、表格、弹层、反馈态必须显式说明选型理由

### 4.6 接口接入

- 先找现有 service 与同领域 adapter
- 优先沿用既有命名与文件结构
- 不在页面中裸写原始接口协议
- 对不确定的字段含义列出待确认项
- 显式处理错误、空值、加载与刷新时机
- 涉及鉴权、拦截器、全局 client 修改时先看 `docs/11_DANGEROUS_AREAS.md`

### 4.7 危险区操作

当改动涉及以下关键词时，必须先查阅 `docs/11_DANGEROUS_AREAS.md`：

`route / router / menu / guard / auth / token / permission / interceptor / http client / shared ui / base component / global style / theme / token / build / config / alias / env / provider / app shell / layout`

禁止：
- 为满足局部页面需求直接重构全局能力
- 未盘点影响面就修改公共默认行为
- 未写回滚说明就动 P0 区域
- 未验证不同角色和主路径就提交危险区改动

### 4.8 排障

- 先分类问题层级，不要直接改第一眼看到的代码
- 先收集报错和请求信息
- 给出"可能根因 → 验证方法 → 修复建议"的链路
- 不要把没有证据的猜测包装成结论
- 修完后列出回归范围和是否需要补文档

## 5. 操作约束（红线）

以下行为在任何场景下都禁止：

- 在未明确业务含义时修改状态字段
- 在未明确分层时直接把逻辑塞进页面
- 直接修改公共基础组件以满足局部需求
- 绕过 service 层在页面里裸调接口
- 根据"类似项目经验"替代当前项目约束
- 在没有明确需求时自行引入大规模重构、新框架、新 UI 库、新路由体系、新请求库

## 6. 输出格式要求

### 6.1 执行任务时

开发前先输出：

```
- 任务目标：
- 参考文档：
- 复用点：
- 改动点：
- 风险点：
- 自测点：
```

开发后补充：

```
- 实际修改文件：
- 与预期差异：
- 遗留待确认项：
- 推荐回归路径：
- 是否需要更新 docs：
```

### 6.2 输出技术方案时

- 所有决策必须给依据，不能只写"建议用 store"，要说明"因为哪些状态跨页面共享"
- 存在不确定信息时，必须写成待确认项 / 临时假设 / 分支方案，不要伪确定
- 不要省略边界场景，至少考虑：空态、loading、error、权限不足、重复提交、数据回填、离开页面后的状态保留或清理

### 6.3 提交前自检

每次提交代码前检查：

- 命名是否符合项目规范
- 类型是否完整（无遗漏的 any）
- 错误处理是否显式
- 空态 / loading / error 是否覆盖
- 权限判断是否到位
- 是否需要更新 docs/

## 7. Skill 引用说明

本项目配套三个 skill，agent 根据任务阶段自动选择：

| Skill | 触发条件 | 输入 | 输出 |
|-------|---------|------|------|
| `skill-tech-solution` | 收到 PRD / 需求，且满足出方案条件 | PRD + docs/ | 技术方案文档 |
| `skill-code-generation` | 收到已定稿技术方案 | 技术方案 + docs/ | 代码 + 测试用例 |
| `skill-frontend-testing` | 代码开发完成 | 代码 + PRD + 技术方案 + docs/ | 自测报告 |

> 各 skill 的详细工作流定义在 `skills/` 目录下。
