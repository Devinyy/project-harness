## 1. 文档目标

本文件不是技术方案模板本身，而是**生成技术方案的行动手册**。

它回答三个问题：

1. 什么场景必须先写技术方案
2. 技术方案应该按什么步骤推导出来
3. 写完方案后如何自检，避免“还没想清楚就开始写代码”

---

## 2. 什么时候必须先出技术方案

满足以下任一条件时，必须在开发前先输出技术方案：

### 2.1 新增页面或完整模块

例如：

- 新增列表页 / 详情页 / 创建页
- 新增审批流、配置中心、策略页等模块
- 新增带有筛选、表格、表单、弹层的综合页面

### 2.2 影响多个层级

例如：

- 同时涉及 route、page、component、service、store
- 需要修改公共组件、页面容器、全局布局
- 需要新增权限、埋点、国际化文案

### 2.3 数据流或状态复杂

例如：

- 多接口串联
- 表单与列表联动
- 草稿态 / 编辑态 / 提交态切换复杂
- 需要 URL 参数、local state、store 共同协作

### 2.4 业务规则多、边界多

例如：

- 角色差异明显
- 状态机复杂
- 字段校验复杂
- 涉及时间、金额、权限、风控等高敏感信息

### 2.5 重构或替换已有实现

例如：

- 旧页面迁移
- 公共组件重构
- 数据流从 local state 调整为 store
- 接口协议变化导致多个页面受影响

---

## 3. 什么场景可以不单独写完整方案

以下场景可不写完整方案，但仍需做最小分析：

- 文案修正
- 样式微调
- 单一低风险 bug 修复
- 不影响数据流的小范围 UI 调整
- 已有成熟模式下的机械性增量改动

即便不写完整方案，也至少要在开发描述中写清：

- 改哪些文件
- 为什么这么改
- 是否复用已有模式
- 风险点是什么

## 5. 产出技术方案的标准步骤

## 5.1 先理解业务目标，不要直接想代码

先回答：

- 用户要完成什么任务
- 当前流程痛点是什么
- 这次要新增的是能力、流程还是展示
- 哪些角色受影响
- 成功的判断标准是什么

如果这些问题没有答案，不要直接写实现方案，应先列为待确认项。

## 5.2 明确范围，不要默认“顺手做掉”

输出方案前，先拆出：

- 本次范围内
- 本次不做
- 依赖其他角色确认的内容
- 后续阶段再处理的内容

不允许把模糊需求直接扩大为重构工程。

## 5.3 先找现有模式，再决定是否新增

对照现有项目，优先寻找：

- 同类页面
- 同类筛选栏
- 同类表单
- 同类列表
- 同类详情抽屉 / 弹窗
- 同类 service / store 组织方式

方案中必须说明：

- 哪些内容直接复用
- 哪些内容参考但不复用
- 哪些内容确实需要新增

## 5.4 决定页面结构

确定以下内容：

- 是否新增 route
- 页面由哪些区域组成
- 是否需要列表 + 详情联动
- 是否需要 Modal / Drawer
- 是否需要保留筛选条件与滚动位置

## 5.5 决定数据流

明确：

- 首屏数据由谁请求
- 查询条件放在哪里
- 表单状态放在哪里
- 派生状态放在哪里
- 何时刷新数据
- 是否需要缓存或共享状态

## 5.6 决定组件边界

必须界定：

- 页面容器做什么
- 业务组件做什么
- 基础组件做什么
- service 做什么
- hooks / composables 做什么
- store 做什么

不要出现“页面里什么都做”或“组件里顺便发请求再改 store 再埋点”的杂糅实现。

## 5.7 最后才落到文件改动与开发步骤

在范围、结构、数据流、组件边界明确后，再写：

- 改哪些目录
- 新增哪些文件
- 哪些文件可复用
- 哪些地方风险最高
- 推荐开发顺序

## 6. 技术方案中的强制决策点

每份技术方案都必须明确以下问题，不能留白：

### 6.1 UI 选型

- 为什么选 Table / List / Card
- 为什么选 Modal / Drawer
- 表单是否复用现有模式
- 反馈态如何设计

### 6.2 数据归属

- 哪些状态用 local state
- 哪些状态进 store
- 哪些状态放 URL
- 为什么这样分配

### 6.3 API 接入方式

- service 是否已存在
- 字段如何映射
- 异常如何处理
- 是否需要 mock
- 是否存在待确认接口问题

### 6.4 复用 vs 新增

- 直接复用哪些组件
- 扩展哪些组件
- 为什么必须新增

### 6.5 风险与回滚

- 最大风险点是什么
- 如果上线后有问题如何回滚
- 是否可通过开关或灰度控制

## 7. 输出技术方案时的推荐顺序

建议按以下顺序组织内容：

1. 背景与目标
2. 范围与非目标
3. 现状调研与复用判断
4. 页面结构与路由
5. UI 与组件方案
6. 数据流与状态管理
7. API 与字段映射
8. 权限 / 埋点 / 国际化
9. 风险与回滚
10. 实现拆分与验收标准

不要一上来就列代码目录和文件名，那是方案的最后部分，不是起点。

---

## 8. 自检清单

技术方案写完后，必须逐项自检：

### 8.1 范围清楚吗

- 是否明确了做什么、不做什么
- 是否列出了待确认项
- 是否避免了范围膨胀

### 8.2 复用判断充分吗

- 是否先查找了现有页面与组件
- 是否解释了“为何新增”
- 是否避免重复造轮子

### 8.3 数据流合理吗

- 请求入口是否清晰
- 状态归属是否清晰
- 是否考虑并发、缓存、重复提交

### 8.4 UI 方案合理吗

- 是否符合 `UI_COMPONENT_GUIDE.md`
- 是否考虑 loading / empty / error
- 是否与项目现有页面风格一致

### 8.5 风险识别充分吗

- 是否识别了公共组件改动风险
- 是否识别了老逻辑兼容风险
- 是否给出了回滚思路

### 8.6 可以直接指导开发吗

判断标准：

- 开发者拿到这份方案后，是否知道该改哪些层
- 是否知道先做什么后做什么
- 是否知道有哪些未确定风险
- 是否知道哪些内容必须复用

如果答案是否定的，方案还不够具体。

---

## 9. 最小合格标准

一份最小合格的技术方案，至少应包含：

- 背景与目标
- 范围 / 非目标
- 页面结构
- 复用与新增判断
- 数据流设计
- API 与字段映射
- 风险点
- 实现拆分

少于以上内容，不视为可开发方案。

---

## 10. 不合格方案示例

以下情况视为不合格：

- 只有一句“按现有页面仿写”
- 只列目录，不讲数据流
- 只说“新增组件”，不说明为何不能复用
- 只写 happy path，不写异常与权限
- 不列待确认项，默认接口和规则都清楚
- 把代码实现细节当作方案，却没有范围与风险说明

---

## 11. 推荐交付物

一份成熟的开发前输出通常包含：

1. 前端技术方案正文
2. 预计改动文件清单
3. 待确认问题列表
4. 风险点列表
5. 必要时附上页面草图 / 数据流图 / 接口映射表

---

## 12. 最终要求

在本项目中，以下行为被视为推荐流程：

1. 先看文档
2. 再做调研
3. 再写方案
4. 再开始编码
5. 编码前回看方案
6. 编码后对照方案做自检

禁止流程：

1. 看到需求
2. 凭经验直接写代码
3. 开发中再补想法
4. 联调时才发现范围、接口、组件边界都没想清楚
